Illumina base call (BCL) data were demultiplexed to sample-level FASTQ files using bcl-convert with a template-generated SampleSheet and explicit runtime parameters captured in the BPM execution context. Input provenance was defined by the run folder (`bcl_dir`) and the resolved sample sheet (locally rendered or retrieved via configured API integration). Lane splitting behavior and output directory structure were controlled by `no_lane_splitting` and `sampleproject_subdirs`, respectively. Demultiplexed FASTQs were written under a dedicated `output/` directory, with optional project/sample subfolder structure when `sampleproject_subdirs=true`.

To support reproducible quality assurance, FASTQ-level quality control was performed with FastQC across all generated FASTQ files, and run-wide quality summaries were consolidated with MultiQC. Optional contamination screening was performed with FastQ Screen when `run_fastq_screen=true`.

Parallel execution used host-derived CPU allocation, with the effective thread budget scaled by `thread_ratio` and subdivided across conversion, compression, and decompression workers for bcl-convert. The exact command-line invocation, selected parameter values, and software versions were recorded in `results/run_info.yaml` for downstream reporting and manuscript preparation.
